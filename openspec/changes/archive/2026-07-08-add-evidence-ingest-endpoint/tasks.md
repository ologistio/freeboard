## 2. Web endpoint and collector auth (feat(web))

- [x] 2.1 Add `CollectorBearerAuthenticationHandler` (new scheme) that parses `v<keyId>.<secret>`, HMACs via `ITokenHasher.TryHashPrefixed`, and looks up `collector_credentials` by token hash. Return `AuthenticateResult.Fail` (yields `401`) for missing/malformed/unknown-key/hash-not-found/key-version-mismatch. For a RECOGNISED credential authenticate SUCCESSFULLY, building a non-human collector principal carrying the collector-id claim, and add an "active credential" claim ONLY when `revoked_at` is null AND (`expires_at` is null OR not elapsed); a revoked/expired credential authenticates but omits the active claim (the `403` is produced later by the ingest policy, not here - a handler `Fail` can only yield `401`). On successful authentication, call `ICollectorCredentialStore.TouchLastSeenAsync` best-effort, swallowing any error so it never fails the request (mirror `BearerAuthenticationHandler`).
- [x] 2.2 Register the scheme and a named ingest authorization policy in `Program.cs` (without changing the default `FreeboardBearer` scheme or `BearerAuthenticationHandler`). Because the collector scheme is NOT the process default, the ingest policy MUST bind it explicitly: call `policy.AddAuthenticationSchemes(CollectorBearerAuthenticationHandler.SchemeName)` (same pattern as the sudo and page-challenge policies), then `RequireAuthenticatedUser()` and require the collector-id claim AND the active-credential claim, so a revoked/expired credential authenticates but is denied -> ASP.NET Core Forbid -> the scheme's default `HandleForbiddenAsync` returns `403`. Expose the ingest policy name as a constant so the endpoint mapping and the route-metadata guard test reference the same value.
- [x] 2.3 Add a dedicated `IngestEndpoint` marker to `ApiRoutes.cs` (distinct from `AuthEndpoint`) and exempt it in `GitOpsReadOnlyMiddleware` (scoped to the marker only).
- [x] 2.4 Add `EvidenceIngestEndpoints.cs`: `POST /api/v1/freeboard/evidence` under the collector scheme. Map it in its OWN route group whose only authorization is `.RequireAuthorization(<ingest policy name>)` (the named ingest policy from 2.2) - do NOT use the parameterless `.RequireAuthorization()` that binds the default session scheme, so the collector bearer token is validated against the collector scheme and not 401'd against the session scheme. Handle the body MANUALLY (take `HttpContext`, do NOT `[FromBody]`-bind) per design D7: enforce the 1 MiB limit (`413`) via endpoint metadata; read the raw body bytes once; SHA-256 those exact bytes for `request_body_sha256`; parse the bytes into a `JsonElement` tree with `JsonDocument.Parse` (or `JsonSerializer.Deserialize<JsonElement>`) inside `try/catch (JsonException)` emitting RFC 7807 `400` ONLY on malformed JSON; then run the semantic validation pass; then project the validated values into internal records and call the store. ONE general rule: do NOT `JsonSerializer.Deserialize<EvidenceIngestRequest>` and do NOT strongly deserialize ANY externally-supplied field into its .NET type - a strongly-typed member throws `JsonException` on a JSON type mismatch (`{"collector_id":123}`, `{"checks":"x"}`, `{"severity":1}`) which surfaces as a framework `400`, violating the invariant that `400` means ONLY a JSON syntax error. Instead read EVERY field from the parsed `JsonElement` tree in the semantic pass and validate its `ValueKind` and value, then project the validated values into the internal typed record(s) (timestamps as `DateTimeOffset`, `metadata`/check `data` kept as `JsonElement`). Validation (all `422`, never `400`): the root is a JSON object; schema_version a present JSON string exactly `freeboard.evidence.v1`; collector_id/run_id present JSON strings <=190; collector_id matches the credential and exists in the register; collector_version if present a JSON string; started_at/finished_at present JSON strings parseable as UTC ISO 8601, with finished_at >= started_at (a missing, JSON-null, non-string, unparseable, or non-UTC timestamp is `422`); `checks` a present non-empty JSON array; each check a JSON object with JSON-string name (non-blank, <=190, unique in run), severity in `{hard,soft}`, status in `{pass,fail,unknown,not_applicable}`; check `data` and top-level `metadata` have `ValueKind == Object` when present. Any wrong `ValueKind` on any field is `422`, not `400`. Snapshot the collector's title/control_id/vendor_id/type; compute the summary counts; build the `{ evidence_id, collector_id, run_id, received_at, hard_fail_count, soft_fail_count, total_count }` body from the `EvidenceAppendResult` for both `201` new and `200` replay (replay uses the stored `received_at`/counts); `409` changed body, `503` store failure; RFC 7807 problem bodies; mark with `IngestEndpoint`.
- [x] 2.5 Add `CollectorCredentialEndpoints.cs`: `POST /evidence-collectors/{id}/credentials` (issue with optional expiry, return raw token once) and `DELETE /evidence-collectors/{id}/credentials/{credId}` (revoke).
